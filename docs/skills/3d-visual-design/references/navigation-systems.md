# Navigation Systems

## Goal

Navigation should lower cognitive load and increase model focus. It should never become the first thing the user has to decode.

## Desktop Pattern

Use a compact but legible system that supports exploration without covering the stage.

Recommended tools:

- Left rail for layers or modes
- Sticky top-right language control
- Quiet reset control
- Optional wheel-based mode or layer switching
- Hidden-by-default insight panel

Desktop should allow:

- Scanning all options quickly
- Hover feedback
- Faster non-reading exploration
- Immediate relationship between navigation selection and camera focus

## Mobile Pattern

Mobile should preserve the model area first and make navigation predictable.

Recommended tools:

- Compact chapter or mode chip
- Fixed bottom dock for primary layer or mode switching
- Small summary card above the dock
- Low-profile about, updates, and audio links
- Deeper controls behind a compact trigger

Mobile should avoid:

- Large button groups in the first viewport
- Permanent transcript or analysis panels in first view
- Tall controls that shrink the model area

## Shared Rules

- Desktop and mobile can differ in control format but should preserve the same content order
- The active item should always be visually obvious
- Navigation should map to model behavior, not just text changes
- A selected item should update focus, labels, or lighting, not only the side panel

## Applied Examples

### Layered chapter

- Desktop: compact left rail
- Mobile: bottom dock with six icons and a summary card

### Mode-based chapter

- Desktop: small vertical or horizontal mode list
- Mobile: fixed bottom mode dock with compact labels

### Deep reading

- Desktop: right-side panel or drawer
- Mobile: expandable card or bottom sheet
